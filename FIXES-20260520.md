## V1 — `firstIntervalId` race on rapid Start → Stop *(Medium)*

**Precaution requested:** guard the `firstIntervalId` callback body.

**Actions:**

1. Added the requested defensive guard at the top of the `firstIntervalId` callback:
   ```js
   firstIntervalId = setTimeout(function() {
     // Defense-in-depth: stop() may have closed ctx between scheduling and firing.
     if (!ctx) return;
     signal = schedule(new Date(t), false);
   }, (delay - (Math.trunc(delay / 1000) * 1000)));
   ```
   This prevents the late callback from dereferencing a closed `AudioContext`.

2. As the primary fix, added an explicit `clearTimeout(firstIntervalId)` to `stop()` so the pending timer is canceled at the source — this also prevents the stale callback from re-populating `signal` (which the visualizer would then draw for one frame after Stop):
   ```js
   if (firstIntervalId) {
     clearTimeout(firstIntervalId);
     firstIntervalId = null;
   }
   ```

Both layers are kept. The guard is the cheap belt-and-suspenders measure; the explicit cancellation is the actual fix.

## V2 — `intervalId` reused for two timer types *(Low)*

**Action:** Split the single `intervalId` variable into two distinct identifiers:

- `setupTimeoutId` — holds the `setTimeout` that fires just before the next wall-clock minute.
- `tickIntervalId` — holds the `setInterval` that fires every 60 s after the first tick.

`stop()` now clears both, using `clearTimeout` for the timeout and `clearInterval` for the interval. The data flow is unambiguous and resistant to "tidying" refactors that would have broken the previous single-variable scheme.

## V3 — DOM lookups not null-guarded *(Low)*

**Action:** Added early-return guards inside the IIFE after each required DOM lookup:

```js
var control_button = document.getElementById("control-button");
if (!control_button) return;
// ...
var nowtime = document.getElementById('time');
if (!nowtime) return;
var canvas = document.getElementById('canvas');
if (!canvas) return;
```

If a host page omits any of `#control-button`, `#time`, or `#canvas`, the IIFE now exits cleanly instead of throwing a `TypeError`. The current `index.html` still provides all three, so no behavior changes for the live page.

## V4 — `new AudioContext()` not wrapped in `try`/`catch` *(Low)*

**Action:** Wrapped the constructor in `start()`. On failure, the UI state is rolled back (`Stop` → `Start`, `play_flag = false`) and the user is told why audio could not start:

```js
try {
  ctx = new AudioContext();
} catch (e) {
  control_button.innerText = "Start";
  play_flag = false;
  alert("Audio could not be started: " + e.message);
  return;
}
```

`alert()` was kept (per the suggested snippet in `VULNERABILITIES.md`) because no dedicated status element exists in the host page. A future improvement is to wire a `<span>` for status messages and replace the `alert` with a DOM update.

## V5 — Unbounded amplitude *(Info — safety)*

**Precaution requested:** do not change the code; add a warning of unbounded amplitude in `README.md`.

**Action:** No code change to `jjy-audible.js`. The gain ramp target remains `1.0`. A new section was added to `README.md` between **Disclaimer** and **Tested environment**:

> ## Warning — audible variant and listening volume
> The "turn the volume up to maximum" guidance above applies only to the antenna-driving variant (`jjy.js`) …
> **Do not raise your system volume to maximum while listening to the audible variant**, especially through earphones or headphones …

The warning makes the volume guidance variant-specific so a future reader who switches `index.html` to load `jjy-audible.js` is not invited to drive a 1 kHz sine at full system volume.

## V6 — Stale leap-second table *(Info — data integrity)*

**Precaution requested:** document explicitly that negative leap seconds are unsupported.

**Action:** Replaced the brief comment above `plus_leapsecond_list` with an explicit note:

```js
// Leap second insersion list in Japan Standard Time.
// Note: negative leap seconds are NOT supported by this implementation.
// There is intentionally no minus_leapsecond_list; the encoder branch for
// a negative leap second (LS1=1, LS2=0) is therefore unreachable. The IERS
// has never announced a negative leap second, so this remains a theoretical
// case in the JJY frame specification.
var plus_leapsecond_list = [
  new Date(2017, 0, 1, 9)
];
```

The encoder still contains the negative-leap-second branch (so the symmetry of the frame layout is visible at a glance), but the comment makes it explicit that the branch is unreachable by design. The list itself was not updated — that remains a maintenance task to be performed against IERS Bulletin C when a future leap second is announced.

## V7 — No `"use strict"` *(Info — defense in depth)*

**Action:** Added `"use strict";` as the first statement inside the IIFE. The existing code is already strict-clean (no implicit globals, no `with`, no octal literals, no `arguments` usage). Strict mode will surface any implicit-global typo introduced by future edits.
